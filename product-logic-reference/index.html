Part 1: Platform Architecture and Infrastructure

Structural foundations only: multi-tenant hierarchy, global vs. local data, module management cascade, access control. Object definitions live in Part 2. Permissions and limits live in Part 3.

1.1 Multi-Tenant Hierarchy

Level Entity Scope Notes
1 Client (Organiser Account) Top container, one per organiser Isolated data, own API domain, own admin credentials
2 Event One edition of a show Shares Client data environment, has local configurations
3 Community (365) Year-round engagement layer Spans multiple Events within one Client

Core rules:

  • RULE: Client status: ON or OFF. OFF disables all credentials, events, and frontends.
  • RULE: Events within a Client share a Global Database. Identities persist across events.
  • RULE: Mobile apps are Single-Event or Multi-Event. Multi-Event uses Event Tags.

1.3 Module Management: The Cascade

Tier Scope Controls
Tier 1: Global MM Client level Frontend modules (by entity type) and backend modules for ALL events under the Client.
Tier 2: Event MM Event level Operates within constraints set by Global MM.

Confirmed structure: Global Module Management lives at /admin/manage (SuperAdmin only, reached from the events list /admin/exhibitions/list), with two tabs: "Modules Managing" (a Frontend table and a Backend table) and "Settings". Event-level control is "Sub/Event Module Management" under Event Setup → Module Management. [872087577]

  • RULE: Frontend role groups in Module Management: Exhibitors, Participants, Buyers, Team Members (Exhibitor is technically not a role). Settings outside a group affect all users. [872087577]

Cascade flow: Global MM → Event MM → Frontend visibility

  • RULE: Disabled at Global MM = disabled on frontend AND toggle disappears from Event MM. Event admin cannot re-enable.
  • RULE: Enabled at Global MM = event admin can disable at event level.
  • RULE: Cascade is strictly one-directional: Global constrains Event.
  • RULE: Cache must be cleared after saving Global MM changes (module flags are cached server-side). [872087577]

Access Control States (per module)

State Behaviour
Open Access Unlogged users see content. Must log in to interact.
Sign-in Only Only logged-in users see the page.

Cascade gotchas

  • If a module is disabled for at least one category/role, it is also disabled for all Guest (unlogged) users.
  • Disabling a module at Global MM removes the toggle from Event MM entirely. Event admins may not know the feature exists.
  • Disabling Buyers: Make a Meeting or Buyers: Messages in Module Management blocks those interactions even if the Permissions Matrix allows them.

Part 2: Platform Objects and Lifecycle

Every core entity: definition, lifecycle states, properties, capabilities, edge cases. Cross-entity interactions live in Part 3.

2.1 Visitor

The base human account. Every person starts as a Visitor. Other roles (Team Member, Speaker, Moderator, Buyer) layer on top.

Aspect Detail
Global properties Name, Email, Company, Job Title, Photo, Login credentials, Behavioural data, External QR.
Local properties Interest categories, Activity categories, T+C acceptance, Notification settings.
Category Assigned during registration, import, API, or Default. Drives permissions, limits, visibility.
Capabilities Browse marketplace, request meetings (regular/online/stand/other), send messages, add sessions to schedule, block time, favourite, book sessions, manage own connection settings, delete own account.

Login and activation

  • RULE: OTP login (Event Setup > General > Settings): enabling the OTP toggle reveals two sub-toggles, SMS and WhatsApp. Email is the default OTP channel; channels can be enabled individually or simultaneously (the user then receives the OTP on all enabled channels). [EP-26859]
  • RULE: OTP validity window is 3 minutes. WhatsApp uses a fixed pre-configured template (no custom text); SMS has a custom text box. Variables available for both: Name, Last Name, Visitor Email, Title, Login, Login URL, OTP Code. [EP-26859]
  • RULE: Badge Scan Login for App — Event Setup > General > Settings toggle (default OFF) + "Categories allowed" (default All) + optional "Check category during Badge Scan Login". Scanning their own badge logs the user in AND activates the account if it was inactive. Fixed errors: account not found → "Your account could not be found. Please contact registration support."; category not allowed → "The badge scanner login function is not available for your category." Button text/colour configured in App Builder. [EP-19419]
  • RULE: Visitor export includes a final "Activation Timestamp" column — set when the account is activated via the activation link OR an API call; format dd-mm-yy hh:mm:ss in the event's local timezone. [EP-26794]
  • RULE: The "Activation Link" variable is available in Marketing > Email Sender (in addition to registration emails); clicking it opens the user's frontend profile and immediately activates it (status → Activated). The link must not work if no profile exists. [EP-27950]

Edge cases

  • A Visitor can simultaneously be Team Member + Speaker + Buyer. Most permissive role wins.
  • Enable Required Fields ON forces redirect to edit profile on login until all required fields are complete.
  • Visitors can manage their own privacy if Allow visitors to manage reception is enabled.
  • Opting out of networking makes the user invisible to all others. Overrides the Permissions Matrix.

2.2 Exhibitor Company

A company-level entity (not a person). Container for company profile, products, brands, team members, stand, sponsorship. Does not attend sessions or request meetings directly. Team Members do.

Settings Layers

Layer Source Override
Category-level settings Inherited from exhibitor category Default
Custom exhibitor-level settings Set per exhibitor Overrides category. Reset All to Category reverts.

Self-service content (exhibitor-created)

  • RULE: Exhibitor-created news require two-level enablement: event-level setting Admin > Registration > Exhibitor registration > Additional settings "Allow exhibitor news" PLUS the per-exhibitor "Allow exhibitor news" toggle in the admin exhibitor profile. [EPLEGACY-2399]
  • RULE: Exhibitor news statuses: waiting for approval / approved / rejected. Approved news appear on the system news page; rejected news never appear. [EPLEGACY-2399]
  • SIDE EFFECT: Editing an already-approved exhibitor news item resets it to re-approval; the OLD version stays live on the news page until the organiser approves the edit. [EPLEGACY-2399]
  • EDGE CASE: Once rejected, the exhibitor can still edit/save the news item but the status remains rejected permanently; the organiser can only delete it. [EPLEGACY-2399]
  • RULE: The organiser can edit exhibitor news pre-approval; "Save" keeps the status, "Save and publish" approves and publishes. [EPLEGACY-2399]
  • RULE: "Additional video" — a second embed-code block on the exhibitor public profile, using the same two-level enablement pattern (event setting "Allow additional video" + per-exhibitor toggle); renders as a separate full-width block under the main profile blocks. [EPLEGACY-2409]
  • RULE: Custom online-room branding: exhibitors get an "Online room settings" frontend tab (event setting "Allow background image" + per-exhibitor toggle "Allow customize room") with background image (aspect ratio 1.8) and text colour; the branding applies to all online meetings initiated by that exhibitor and/or its team members. Sessions get the same background image + text colour pickers in session edit when "online session" is on. [EPLEGACY-2405]

Edge cases

  • Meet button only appears if at least one team member is available (checked against permissions, opt-in, slots, limits).
  • Disable meetings/messages without Team Members ON: exhibitors without TMs cannot send/receive. Existing chats/meetings created before continue.
  • Hide exhibitors without team members ON: hides them plus products and brands. Sync up to 15 minutes.
  • Exhibitor registration does not pre-populate from existing accounts with the same email.
  • Exhibitor-to-Sponsor conversion creates a separate entity. Changes no longer sync.
  • Online/Onsite meeting settings on exhibitor page: restriction source is always Category, cannot be changed to Custom.
  • RFP setting: Category-only control. No custom override possible.
  • After setting a new category, settings with existing custom values are NOT overwritten.

2.3 Exhibitor Team Member

A person linked to an Exhibitor. Simultaneously a Visitor (personal account) AND a Team Member (exhibitor context). Acts on behalf of the Exhibitor.

Key Rules

Rule Detail
Category inheritance Inherits exhibitor category of parent Exhibitor. No independent exhibitor categories.
Limits Meeting/chat limits inherited from exhibitor category. No separate custom limits.
Connection settings Meetings Allowed, Chats Allowed per TM. Bidirectional sync between exhibitor and TM.
Locality Local to each event. Must be assigned per event.
Headcount limits Set per exhibitor category and/or per individual exhibitor.
Creation form 6 default fields (Name, Last Name, Phone, Email, Role, Job Title) + visitor fields with Use for Team Member Creation checked.
  • CONSTRAINT: One email = one exhibitor. A team member email can be assigned to only ONE exhibitor company profile; managing multiple companies requires separate logins or a parent-child structure. [EP-47989 context]

Removal cascade. Removing a TM from an exhibitor reverts them to default participant category AND removes them from all chats/meetings of that exhibitor (confirmed v10 behaviour).

  • RULE: Detach via API (/account/set with both exhibitor_id and exhibitor_external_id = 0/"0") detaches the team member; cascade: removed from the exhibitor's TM lists (frontend, admin exhibitor profile, Exhibitor Manual), removed from the Share-My-Profile dashboard image, and removed from ALL their meetings and chats. Idempotent if already detached; applies to the current event only (detach-from-all-events was not implemented). [EP-35903]
  • RULE: Admin exhibitor profile → Team Members table: Email is merged with First & Last Name into a single "Team Member" column; the Job Title column also shows Company Name; new Phone Number column added. [EP-44527]
  • EDGE CASE: If exhibitor disables TM's meetings/chats via connection settings, TM gets a notification. TM can re-enable themselves (sync is bidirectional).
  • EDGE CASE: If Allow exhibitors to manage connection settings of TMs is OFF, import values in Meetings Allowed and Chats Allowed columns are ignored.
  • EDGE CASE: Error on confirming meeting with restricted TM: "You can't confirm meetings with this user because of their connection settings."

2.4 Speaker

Role assigned when a Visitor is added to Management > Sessions > Speakers. Always also a Visitor. Elevated capabilities in session rooms.

  • EDGE CASE: Hiding Favourite for Participants also hides it for Speakers (shared participant base).
  • EDGE CASE: Role-level settings take precedence over category settings.
  • EDGE CASE: Session Visible to restrictions do NOT apply to Speakers. They always see their assigned sessions.
  • EDGE CASE: Speaker profile shows both regular sessions AND exhibitor events they speak at.

2.5 Moderator

Role assigned when added to the Moderators list. Highest control in session rooms. Always also a Visitor.

Capabilities

  • Enable/disable participant microphones, cameras, screen sharing
  • Q+A management: respond, publish, dismiss
  • Poll management
  • Attendee role management
  • Green Room access (Livestream)
  • Start/stop broadcast (Livestream)

  • View analytics: viewer count, engagement

  • EDGE CASE: In Breakout: participants can only enable camera/mic if Moderator grants access. Without moderator action, all are muted/camera-off by default.

  • EDGE CASE: Participant logged in on multiple devices may appear as "Undefined" in the participants list.
  • EDGE CASE: Session Visible to restrictions do NOT apply to Moderators.

2.6 Sessions

Organiser-created content events. Scheduled presentations, panels, workshops, or other content activities.

Key Properties

Property Group Fields
Taxonomy Tags, Tracks, Types
Visibility Visible to (category restriction), Make visible to all toggle (view but restrict interaction)
Booking Pricing (paid sessions), Schedule limit (max bookings)
Status flags Active/Inactive, Featured, Critical
Internal fields Notes (backend-only), Flag as Critical
Room settings Disable video/mic by default, Disable raise hand, Disable participant count, Disable Q+A, Disable polls, Disable chat, Custom text colour and background
  • RULE: Sessions (and Exhibitor Events) carry an internal "Notes" field — backend-only, never shown on the frontend; included in session export (Notes column) AND session import at Data > Import/Export. [EP-29190]
  • RULE: "Flag as Critical" checkbox (default unchecked), backend-only; visual indicator in the session list; filterable in session management; included in session export and import. [EP-29221]
  • RULE: Creating a session that conflicts with another (overlapping time or location) raises a warning popup, on frontend and backend scheduling alike; the user may ignore and proceed, or adjust. [EP-16924]

Module split and admin structure

  • RULE (supersession): Sessions and Exhibitor Events are now SEPARATE modules. Sessions live under Management > Sessions; Exhibitor Events under Meeting Formats > Exhibitor Events. Each has its own independent Types, Subject Tags and Tracks — the Exhibitor Events taxonomies were CLONED from Sessions at the Meeting Formats restructure and are independent thereafter (Sessions taxonomy changes no longer propagate). This supersedes the earlier rule that Exhibitor Events share Tracks/Types/Tags with Sessions. [EP-49369, EP-48239]
  • RULE: Admin list filters: the Sessions page gained filters Session Type, Subject Tags, Tracks, Speaker, Moderator, Location; the Exhibitor Events page gained the same six PLUS Custom Questions (exhibitor-event custom questions become filters) and an Exhibitor/Created-By filter. [EP-48239]

Invite links and deeplinks

  • RULE: Direct invite link — auto-generated per session after first save (Management > Sessions add/edit). Using it books the session and redirects to My Schedule. Exhibitors cannot use invite links; impersonate mode cannot use them. Access/visibility/payment/limit restrictions still apply everywhere EXCEPT My Schedule once booked via the link. [EPLEGACY-2408]
  • RULE: Meeting "Copy invite link" exists only on Confirmed, future meetings; the joiner is added to Additional participants (a member of one of the two main parties joins their owner's side, preventing duplicate presence). The link is invalid if the meeting is rescheduled/cancelled/past, and blocked if the joiner's timeslot is occupied. [EPLEGACY-2408]
  • RULE: Deeplinks for Sessions and Exhibitor Events: session create/edit gains a read-only system-generated URL field (mirrors Direct invite link logic; includes the deeplink variant when deeplinks are enabled for the event). Sessions export gains a "URL" column. Open behaviour: app installed → session page in app; not installed → web. Not-logged-in app open → login prompt then redirect to the session. No permission / nonexistent session / Sessions disabled in app → redirect to main page with "Sorry, this page is not accessible." Links honour session visibility/permissions. [EP-44893]

Reminders and ratings

  • RULE (supersedes hardcoded behaviour): Session/exhibitor-event reminder times are a configurable multiselect at /admin/events/config → General Settings ("Notifications Before Session/Exhibitor Event Start (mins)"), default 10, 15, 30, 60; values 0–60, no duplicates; no values selected = no reminders. Replaces the previously hardcoded 60/30/15/10. Applies to web AND app, only for events in the user's schedule. A given reminder is sent once; if the session start changes after a reminder was sent, it is re-armed against the new time. [EP-36743]
  • RULE: Session ratings exist on web and app: after a session ends the user can rate (stars + comment) or mark "Session did not take place"; ratings are editable afterwards. Entry points in app include an automatic post-session prompt, a rate-list page, the session page button, and an App Builder menu block. [EP-19438, EP-33802]
  • CONSTRAINT: Speakers and Moderators cannot rate sessions/events. Team Members cannot rate their own exhibitor's sessions (bias prevention). Rating exists only for sessions and events — no other content types. [EP-33802]

Schedule limits are client-side only. Session schedule limits run as client-side JavaScript. No server-side validation. Advanced users can bypass.

  • EDGE CASE: Breakout hard cap: 250 participants.
  • EDGE CASE: On-Demand: invisible on frontend after the To date passes.
  • EDGE CASE: Make visible to all ON: all users see session, but only specified categories can add to schedule. Tooltip can explain restriction.
  • EDGE CASE: Removing paid session from schedule: impersonate participant via Management > Participant.

2.7 Exhibitor Events

Sub-events created by exhibitors within the organiser's event. Layered enablement, a submission approval status machine, and an optional participant application/approval process.

Enablement Cascade

Level Where Detail
1 Module Management enables Exhibitor Events Toggle turns the functional on/off. [EP-931]
2 Exhibitor Category "Allow host Exhibitor Events" (default OFF) per category. [EP-931]
3 Individual Exhibitor Same settings overridable per exhibitor. [EP-931]
  • CONDITION: Exhibitor category settings (also settable per individual exhibitor): "Allow host Exhibitor Events" (default OFF); limit of created Exhibitor Events; "Blocks timeslot" (default OFF — whether exhibitor events affect timeslot availability); "Auto Approve Exhibitor Events" (default OFF, shown only when Exhibitor Events are on — auto-approves submissions and changes). [EP-931]
  • EDGE CASE: Submission auto-approval triggers at 3 independent levels: per category, per session type, per exhibitor. Any one being ON triggers it.

Submission lifecycle (status machine) — CORRECTED

Full status set: New → Approved / Rejected / Edited / Changes Rejected, plus terminal Rejected for Security Concern. (Supersedes the earlier "4 approval states" description.) [EP-931, 726990849]

  • RULE: A newly submitted event has status "New" — visible only to the creating exhibitor (frontend profile section) and the organiser (admin); not on the public profile or Sessions page until approved. Editing a "New" event keeps status "New". [EP-931]
  • RULE: Rejecting a "New" event → "Rejected" (last info kept); the exhibitor can re-edit → status "Edited" → can be approved/rejected again; rejected again → "Rejected". [EP-931]
  • RULE: Editing an already-approved event ("Approved" or "Changes Rejected") sets status "Edited" and requires re-approval. Frontend users keep seeing the LAST-APPROVED version until the changes are re-approved. (Supersedes the earlier statement that only the organiser can edit after approval — exhibitors CAN edit, triggering re-approval.) [EP-931]
  • RULE: If changes are rejected, the event reverts to last-approved info for exhibitor and organiser, status "Changes Rejected"; users can still see and sign up for the old version, and it stays in their schedules. [EP-931]
  • SIDE EFFECT: Auto-expiry — a "New" event whose end datetime has passed flips automatically to "Rejected"; an expired "Edited" event flips to "Changes Rejected". [EP-931]
  • RULE: Statuses are visible only to admins and the creating exhibitor, never to frontend users. [EP-931]
  • RULE: "Save without approval" — third action button on event review (between Reject and Approve & Save): saves organiser edits (e.g. internal Notes, Critical flag) without changing the approval status; the event stays invisible on the frontend until Approved. Available for any status. [EP-32197]
  • RULE: Event creation date/time and last-edit date/time are logged (recorded for exhibitor-made changes only) and appear as "date of creation" / "date of last edit" columns in the Session export. [EP-31429]

Permanent rejection. Rejected for Security Concern is permanent. No edit, resubmit, or appeal. The organiser picks a rejection reason from a predefined, editable list (Admin > Management > Session > Session Config); the reason is shown to the exhibitor on the frontend event card. [726990849]

Submission requirements and content safety

  • RULE: The organiser can attach a custom question form to exhibitor event submission (reuses session-registration custom forms). Exhibitors must answer during event creation; answers display READ-ONLY at the bottom of Event details in admin; the organiser approves/rejects based on the answers. [EP-26093]
  • RULE: Policy compliance — "Ask for Policy compliance" + "Content Submission Policy" textbox (Session Config, default OFF): the exhibitor must tick a Policy checkbox (Policy opens on a separate page) before saving the event submission. [726990849, EP-26093, EP-31429]
  • RULE: Offensive-content reporting (Session Config, all defaults OFF): "Allow Report Offensive Content" lets frontend users report exhibitor events; the organiser manages a predefined report-reason list (CRUD); "Allow Exhibitors to see reports" exposes report counts + details to the exhibitor WITHOUT reporter personal data and with no download. [726990849]
  • SIDE EFFECT: A Session email template notifies the organiser of reports, batched every 15 minutes (no email if no reports in the window). On marking "Rejected for Security Concern", the organiser can tick a checkbox notifying all reporters of the removal. [726990849]
  • RULE: The /admin/events list gains a "Report" column (count of reporters per event); the Session export gains columns "Exhibitor event status" and "Offensive content reports". [726990849]
  • RULE: Status-change notifications: organiser emails on submission and on changes to approved events; exhibitor emails on approval, approval-of-changes, rejection, security rejection and rejection-of-changes (same set as bell notifications). Platform notifications are gated by a toggle at /admin/management/session/config (default OFF); emails are configured under Session → Templates. [810450952]

Participant application / approval process

  • CONDITION: Optional per-event approval: "Enable Approval Process" toggle on Exhibitor Event create/edit (default OFF); "Application Deadline" date+time (required when enabled; after it users can no longer apply); "Autoapproval" toggle (default OFF). [EP-39836]
  • RULE: With approval on, users see "Submit Request"/"Apply" instead of Add to Schedule. The event appears in the user's schedule immediately on application with status "Waiting for Approval" (blocking its timeslot if the event type blocks); approved → scheduled; rejected → removed from Schedule with no re-apply. Removing an approved event triggers a confirmation modal warning that reapplying requires approval again. Applying to a timeslot-blocking event while a conflicting time-blocking item exists is prevented with an error. [EP-39836, EP-39841]
  • RULE: Deadline autoapproval — event with a capacity limit → approve first-come-first-served until full, rest rejected; no limit → approve all pending. Respects blocked timeslots (a user whose schedule conflicts is rejected). [EP-39836]
  • CONSTRAINT: Approval cannot coexist with an unlimited Schedule limit: saving with "Enable Participant Approval" ON and Schedule limit = 0 is blocked ("Unlimited participation is not allowed together with the Approval process…"). On-Demand exhibitor events hide the limit field, so the approval toggle is hidden for them too. [EP-46203]
  • RULE: Application page — a tab on the Exhibitor Event edit page, reachable from the pending-applications indicator on the events list. Application statuses: Pending, Approved, Rejected, Autoapproved, Autorejected. Approve/Reject buttons exist only for Pending; actions are one-time and irreversible. Per-applicant insights: Exclusive Attendance Yes/No (only exhibitor event in their schedule at that time) and Conflicting Events count + titles. The organiser may manually approve beyond the participant limit (warning modal). [EP-41663]
  • RULE: Attendees tab — lists users with the event in their schedule (approval-process events show approved users only); the admin can add/remove users and toggle Attended without resaving the event. Attended is set when: the user is scanned in via the check-in app; OR an Exhibitor scans the badge of a user who has that exhibitor's event scheduled, during its active time INCLUDING a 1-hour window before and after; OR the admin marks it manually. Attended also surfaces in the exhibitor view (Total Attended counter, Attendance Status filter) and the leads file. [EP-41665, EP-39837]
  • RULE: Approval notification: "Application Approved" notification + email template (Management → Sessions → Templates). CHANGE: ALL rejection notifications (platform + email) and rejection indications on cards were subsequently REMOVED — users receive nothing when rejected, whether manually or automatically. [EP-41666, superseded in part by EP-46207]
  • EDGE CASE: Disabling the Approval Process mid-flight triggers the autoapproval mechanism; users can then add the event directly. Re-enabling later: existing schedule holders keep Approved state; new users must apply. [EP-39836]
  • RULE (exhibitor view): An approved event with approval enabled shows "Organizer manages approval process", the application deadline, and an Approved Participants counter/list. [EP-39836]

Algorithmic batch auto-approval (supersedes scheduled auto-approval)

  • RULE: "Initiate auto-approval algorithm" button at Sessions Config (with confirmation popup): the organiser selects Exhibitor Events to process (events WITHOUT a limit are disabled in the selector with an explanatory tooltip) and runs the algorithm on demand. The previously released scheduled per-event auto-approval was REMOVED; manual approval remains. [EP-46110, EP-46201]
  • RULE: Algorithm behaviour: respects blocked-slots logic; maximises total approvals; ensures every user gets ≥1 approval if they have none; never approves overlapping events for one user; balances approvals across users; under-subscribed events (applications < limit) approve all. [EP-46110, EP-46204]
  • RULE: Re-runnable: approved users keep their status; algorithm-rejected users may be approved in later runs; MANUALLY rejected users stay rejected. [EP-46110, EP-46204]
  • RULE: Exclusions: events with status Rejected or Rejected for Security Concern are not processed (Approved/Edited/Changes Rejected are); On-Demand-type exhibitor events are excluded. [EP-46204]
  • EDGE CASE: No exhibitor events with approval enabled → "There are no Exhibitor events with the approval process enabled."; events exist but all Application Lists are empty → "There are no Exhibitor event applications to process." [EP-46201]

Settings, taxonomy and creation rules

  • CONDITION: Settings group "Exhibitor Events" (now under Meeting Formats > Exhibitor Events after the restructure): Allow Speakers / Allow Moderators / Allow Sponsors (default OFF; sponsors = system Sponsors); Allow Online / Allow Hybrid (default OFF); Allow other location (default OFF). Part 3: Permissions, Limits and Interaction Logic

Single source of truth for "Can X do Y to Z?" Covers the Permissions Matrix, the five-step evaluation chain, complete meeting/chat limits, override hierarchy, and exception rules.

3.1 The Five-Step Evaluation Chain

Every user-facing interaction (meet, message, view, favourite) passes through these checks in order. ALL must pass. Any single denial blocks the action.

Step Check Question
1 Module Management Is the module enabled at Global MM AND Event MM?
2 Permissions Matrix Is initiator's category/role allowed to act on receiver's category/role?
3 Networking Opt-In Has receiver (and initiator) opted into networking?
4 Connection Settings Has receiver (or their exhibitor) restricted this interaction?
5 Limits Has initiator or receiver exhausted daily or total limits?

How the chain is enforced at runtime (Meet button)

  • RULE: On WEB, when a user clicks "Meeting" on an Exhibitor-with-TMs / Pavilion-with-TMs card or profile (or related Product cards/profiles), the system runs the availability checks ON CLICK — the button is NOT pre-hidden. The pre-hide approach was dropped after load testing (decision revised 18.02.2025); target check time is under 1 second. [EP-35513]
  • RULE: Checks executed: Permission Matrix allows meetings; team member is opted into networking; exhibitor has not turned meetings off for that team member; team member has not disabled meetings in individual connection settings; a meeting limit is still available. [EP-35513]
  • RULE: If ANY web check fails, the error shown is: "There is currently no availability with this exhibitor. Please try messaging the exhibitor for more information." If all pass, the user proceeds to the Meeting request page. [EP-35513]
  • RULE: On the MOBILE APP, the same checks (Permission Matrix incl. parent exhibitor category, Networking Opt-in, exhibitor per-TM setting, individual connection settings, Meeting Limits) instead HIDE the "Meeting" button on exhibitor/product cards and profiles when no team member passes. Same checks for any user type. [EP-37473]

3.2 Permissions Matrix

Configured at Event Setup > Networking and Matchmaking > Manage Permissions. Y-axis (left column) = Initiator ("who"), X-axis (top row) = Receiver ("with whom"). Three toggles per intersection: Meet, Message, View Profile. "Guest User" = a user not logged into the event. [697434241]

  • RULE: Matrix permissions are ONE-WAY: cat1→cat2 is independent of cat2→cat1. A blocked side can still respond in an existing chat and accept meetings initiated by the allowed side. [EP-56]
  • RULE: View Profile OFF = users in that category are hidden from listing pages (toggling it off also hides the object's listing from the respective listing page for that category). [697434241]
  • RULE: If NOT allowed for at least one category/role, also NOT allowed for Guest (unlogged) users — most-restrictive rule for unlogged visitors. [EP-56, 697434241]
  • RULE: Multi-role users: permissive union (OR). If any one role/category permits, the action succeeds; prohibited only if ALL roles/categories prohibit it. [EP-56, 697434241]
  • RULE: The matrix supports bulk row/column on/off controls per interaction type. [EP-4168]
  • SIDE EFFECT: Implementing the matrix REMOVED these legacy settings — if older documentation lists them as active, they are obsolete: category-level "hide Messages/Meetings" functions, exhibitor "Restriction of searching each other", "Not allow Buyer to Buyer", and the Module Management "Make a Meeting" permissions (local and global). [EP-56]
  • EDGE CASE: View Profile OFF for participant to exhibitor: participant cannot see the exhibitor profile, but CAN still see that exhibitor's products.

3.3 Override Hierarchy (Priority Order)

Priority Rule Effect
1 Module Management OFF Overrides everything. Disabled = no permission can enable it.
2 Networking Opt-Out Overrides Permissions Matrix. User who opted out is invisible.
3 Connection Settings Overrides Permissions Matrix for the specific user.
4 Permissions Matrix Default when no overrides apply.
5 Limits Final gate. Exhausted limits block even fully permitted actions.

Networking Opt-In data rules

  • RULE: Networking Opt-in is a predefined registration field (Registration Settings > Visitor > Registration Pipeline > drag "Networking Opt-in"). Its DEFAULT value is NO — every account created via external registration or API is opted OUT of networking unless the field is explicitly set. Import value 1 = opt-in. [736067609]
  • RULE: "Enable inversion for networking opt-in" (Registration Settings > Visitor > Additional Settings) flips the question to opt-OUT phrasing. SIDE EFFECT: enabling inversion INVERTS ALL EXISTING participant data; with inversion ON, import value 1 = opt-out. [736067609]
  • CONSTRAINT: At database level the value is stored invariantly: Opt-in = 1, Opt-out = 0, regardless of the inversion setting. [736067609]

3.4 Badge Scan Exception (Mobile App)

Badge scan bypasses the permission matrix. Once two users have scanned each other, normal restrictions no longer apply.

  • RULE: Badge scan adds the scanned user to the scanner's list, and the scanner ignores ALL permission restrictions toward the scanned user (view / message / meet). Chat and meeting buttons always appear. [EP-56]
  • RULE: Mutual scan: messaging and meetings permitted regardless of GDPR opt-in.
  • RULE: Existing chats: messaging remains enabled regardless of subsequent GDPR or permission changes.
  • RULE: Favouriting after badge scan might bypass permission restrictions.

3.5 Chat (Message) Limits

Global Hard Limit

Limit Value Scope
Initiated chats per day 300 Per account, hard cap, cannot be overridden

Applies to: Exhibitor company accounts, Team Members, Visitors, Buyers.

Applies only to creating new chats. No limit on messages within existing chats.

  • RULE: The 300/day limit is specifically a NEW-CHAT-CREATION rate limit. It is enforced via sockets (backend pushes the limit; web/iOS/Android apply it automatically), applies to everyone on all events by default, and has NO admin enable/disable. [EP-35014]
  • RULE: The 24-hour window counts from the FIRST message sent. On exceeding, the user is blocked from creating new chats with the warning: "You are creating new conversations too quickly and have reached the limit. Please try again in __ hours and ___ minutes." [EP-35014]
  • EDGE CASE: Chat-initiation limits do not apply when a chat with that user already exists. [EP-24046]

Value Interpretation (Chat and Meeting Limits)

Setting Value Meaning
Toggle OFF Unlimited
0 Blocked — silently (most common misconfiguration on the platform)
N Capped at N
  • EDGE CASE: A chat limit of 0 causes a SILENT failure: messages appear to send on the sender's side but are never delivered, with no error shown. The Opt-in Speed Networking buyer meeting/chat limits also default to 0 and fail silently if not set at event setup. [697401523, 809893924]

Exhibitor vs. Team Member Chat Limits

  • RULE: Chat limits POOL at the EXHIBITOR ENTITY level: the exhibitor company account and ALL its team members draw from one shared pool. Per-team-member chat limits are not supported. [697401523]
  • RULE: TM chat limits come from the TM's participant badge category first: if the team member's badge (participant) category has its own limits set, those apply INSTEAD of the exhibitor pool. [697401523]
  • RULE: If the TM badge category has no limits set, the exhibitor category limit applies as fallback.

Setup: Exhibitor "Gold" = 20 daily chats. 3 TMs in "Exhibitor Staff" category.

  • Case A: "Exhibitor Staff" toggle OFF. TMs fall back to the exhibitor pool. 20 chats shared across company account + 3 TMs.
  • Case B: "Exhibitor Staff" daily limit = 10. Each TM has own limit of 10 (from participant category). Exhibitor company has separate limit of 20.

Frontend Behaviour (Chat Limit Reached)

  • User redirected to community page with popup: "You used your chats initiations limit!"
  • Existing chats continue to function. Only new chat initiation is blocked.

3.6 Meeting Limits

Configuration Locations

Scope Path
Exhibitor categories Registration Settings > Exhibitor > Categories > gear icon
Participant categories Registration Settings > Visitor > Categories > gear icon
Individual exhibitor override Exhibitor profile > Settings tab (takes precedence over category)

Onsite/Offline and Online meeting limits are configured independently with the same four settings each. Value interpretation matches chat limits.

  • RULE: All 8 meeting-limit fields (daily/total × pending/confirmations × onsite/online) exist identically for exhibitor AND participant categories, including the default categories. [697401494]

How limits are counted (counting sides)

  • RULE: Two limit kinds per category: "Pending requests limit" (simultaneous pending meetings) and "Confirmations limit", each in daily and total variants, separately for online/offline. [EP-24046, 697401494]
  • RULE: PENDING limits count against the INITIATOR side. CONFIRMATION limits count against the RECEIVER side. [EP-24046, 697401494, 767885647]
  • CONDITION: Autoconfirm ON → pending limits are NOT applied to either side (no meeting is ever in pending state); the confirmation limit is still counted against the receiver. [EP-24046, 697401494, 767885647]
  • CONDITION: Autoconfirm OFF → pending limit counted on the initiator side; confirmation limit on the receiver side. [EP-24046, 767885647]
  • RULE (compliance fix, Nov–Dec 2025): Category-level meeting CONFIRMATION limits are enforced even with auto-confirm ON — auto-confirmed requests stop once the limit is reached. This does not change the rule above that pending limits are skipped under autoconfirm: pending and confirmation limits are different counters. [1462075393]
  • EDGE CASE: Rescheduling a pending OR confirmed meeting never re-applies category limits. [EP-24046, 697401494, 767885647]
  • RULE: Table (concierge) meetings use the same limit logic as regular onsite/offline meetings. [EP-24046, 697401494, 705233031]
  • RULE: When a pending limit is reached, the corresponding location type (online/offline) is HIDDEN from the meeting form's Location field. [EP-24046]
  • EDGE CASE: If both the total AND daily limit of one meeting type are reached, only the TOTAL-limit error is shown. [EP-24046]
  • RULE: Online "minutes credit" exhaustion does NOT block creating online meetings; it hides the "Join room" button and kicks the user from the room when the credit runs out mid-meeting. [EP-24046]
  • RULE: Meeting and message category limits are enforced identically on the mobile app and web, including the full error mechanism. [EP-24046, 767885647]
  • ERROR TEXT: "Some users you are requesting a meeting with have exceeded the online/offline confirmation limit." — the request/Next-step button is disabled. [697401494]

Meeting Auto-confirm (global and category level)

  • RULE: Global "Meetings autoconfirm" lives at /admin/appointments. A participant-category-level "Meeting autoconfirm" toggle also exists in the category settings pop-up: it auto-confirms requests CREATED by that category (initiator side). [705233031]
  • RULE: The global /admin/appointments autoconfirm SUPERSEDES category-level autoconfirm rules. [705233031]
  • RULE: A reschedule is auto-confirmed only if it is initiated by the autoconfirm-enabled side. [705233031]
  • RULE: With category autoconfirm, the Pending status filter is hidden only after existing pending meetings are accepted/cancelled — existing pending meetings remain visible until resolved. [705233031]

  • RULE: Meetings Auto Expire: auto-cancels pending meetings after set time. Frees pending limit slots.
  • RULE: Block time slots for pending meetings ON: pending/incoming timeslots blocked for new meetings.
  • RULE: Auto-complete outside meeting time: badge scan marks meeting Complete regardless of timing.
  • RULE: Meetings created by organiser via Meeting Wizard are automatically Confirmed.

Blocked time and schedule interaction

When OFF, the Add/Edit blocked time pop-up disables dates/times outside event dates and the picker autofocuses the nearest event date; applies to BOTH exhibitor (incl. managing team members' blocked time) and participant blocked time. Existing blocked times persist when the toggle flips or event dates change. [EP-34594]
  • EDGE CASE (overlap asymmetry): An Opt-in Speed Networking session card MAY be laid over existing blocking schedule items (users can be enrolled while already busy), but no new blocking item can be added over the Opt-in SN session. [EP-30302]

Meetings minimal requirements (per category)

  • RULE: "Meetings minimal requirements" — per-category setting "Required number in total" (positive integer; 0/empty not allowed) in visitor/exhibitor category settings. Users see the disclaimer "Your confirmed meetings: X / Required meetings: Y" on My meetings / Team meetings. CONSTRAINT: failing the requirement triggers NO sanctions. [EP-23256]

3.9 Limits Troubleshooting Checklist

When meeting or message buttons do not work, check in this order:

  1. Permission Matrix (correct categories on X and Y axis? Remember permissions are ONE-WAY — check the initiator's row [EP-56])
  2. Module Management (Meetings/Messages enabled at Global AND Event level?)
  3. Daily and Total chat/meeting limit settings (value = 0? Note: 0 fails SILENTLY for chats — messages look sent but are not delivered [697401523])
  4. Which side's limit is exhausted: pending limits hit the INITIATOR, confirmation limits hit the RECEIVER; with autoconfirm ON only confirmation limits apply [EP-24046, 1462075393]
  5. Missing online/offline option in the meeting form's Location field = that side's pending limit is reached [EP-24046]
  6. Table meetings toggle (if applicable)
  7. Connection settings (exhibitor or TM disabled meetings/chats?)
  8. Networking Opt-in (user opted out? Accounts created via API/external registration default to opted OUT [736067609])
  9. Chat pool: exhibitor + team members share ONE chat pool unless the TM's badge category sets its own limits [697401523]
  10. New-chat rate limit: 300 new chats per 24h from first message, socket-enforced, no admin override [EP-35014]
  11. Online meetings: user can CREATE meetings with exhausted minutes credit but cannot JOIN the room [EP-24046]
  12. On web the Meet button shows an error only on click; on the app it is hidden — absence of the button on app vs error on web can be the SAME root cause [EP-35513, EP-37473]

15.2 Badge Scan Flow

Mobile app scans badge → attempts local resolution (barcode to account ID) → if found, uses account ID (fast path) → if not found, sends raw barcode to backend (slow path) → backend searches barcode types (barcode_new, external_qr, external_barcode, external_code).

Critical edge cases

  • Barcode collision: if a barcode value equals another user's Account ID, the scan attributes to the wrong person.
  • Account ID overflow: max value 2,147,483,647 (32-bit integer). Barcodes exceeding this cause overflow errors.

Part 19: Backend — Engine & Services Logic

Server-side internals: the matchmaking / recommendation pipeline, the meeting-generation engines, the email/SES pipeline, badge-scan resolution, floor-plan rendering, integration sync and retry behaviour, caching, and the hard system limits that constrain all of the above. This is the "how it actually runs" layer beneath the product rules in Parts 1–17 and the admin surface in Part 18.

19.1 Matchmaking and Recommendation Pipeline

  • ENGINE: "Enable Matchmaking" drives a Data Science real-time recommender pipeline with processing statuses In Progress / Error / Completed. Turning matchmaking OFF calls a lambda to FREE the allocated resources. The flag is reset to OFF on event create/clone. [EP-34318]
  • ENGINE: The recommender connects Interest categories (users) to Activity categories (content/exhibitors). Multi-Space mode groups categories into spaces and weights by spaces-matched first, then categories-within-space. [Part 7.1]
  • RULE: "Recommended for you" sorting interplay: disabled during Elastic word search; a match score of 0 → random order; hidden for unlogged users; sponsored products inherit their exhibitor's MM rank. The legacy esettings "Matchmaking sorting" control was removed. [EP-27225, EP-27246]
  • RULE: Onboarding completion dispatches a "recommendations updated" notification ~5 minutes after completion; the notification writes selections to the user's Interest categories (the interest-category limit is ignored during onboarding). [EP-40162, EP-40531]
  • RULE: Like propagation — a "like"/favourite can propagate across all team members of an exhibitor via an opt-in API parameter. [EP-50840]
  • EDGE CASE: Empty recommendation blocks (email or frontend) almost always mean the user has NO interest categories populated. [Part 7.1, Part 12.1]

19.2 Meeting-Generation Engines

Three distinct generation engines with different logic. They must not be confused.

Speed Networking — Walk-in (Check-in)

  • ENGINE: Max meetings per user is auto-calculated from session geometry: N × meeting_length + (N − 1) × break_length ≤ session_duration. [EP-26344]
  • RULE: Live match voting feeds relevance before schedule generation; tables are manually editable; a Public Meeting Display ("airport board") shows assignments. The minimum gap between "Send Final Schedule" and "Meetings Start" was reduced from 5 minutes to 1 minute. [EP-26344, EP-30282]

Speed Networking — Opt-in (pre-matched)

  • ENGINE: Participation conditions are built from registration fields flagged "Use for Speed Networking". Auto-enrolment runs HOURLY. A two-sided visibility matrix controls who can be matched with whom (recommended templates: Open Networking / Buyer-Focused / Structured Access). [EP-26368, EP-30282]
  • RULE: The Meeting Sponsor role and ALL sponsored-meeting logic were REMOVED from Opt-in SN (they exist only in the Meeting Program). Sponsored-limit validation applies only to newly created meetings. [EP-30282, EP-39133]

Meeting Program (rebuilt 2025; frontend "Meetup")

  • ENGINE: Generation uses MUTUAL relevance only, in four strict tiers: SR↔SR → SR→R → R→SR → R↔R (SR = super-relevant, R = relevant). The matchmaking SCORE does NOT influence generation. [EP-41245, EP-43483]
  • RULE: One pairing per company (companies matched by NAME). Sponsored meetings are generated BEFORE free meetings. The 1st generation now ALSO generates free meetings (previously sponsored-only). Hosted Buyers never receive free meetings. Sponsored Team Members share the exhibitor's sponsored allocation (default 30; 0 = none; toggle off = unlimited; category-level only). [EP-41245, EP-43924, EP-49069]
  • RULE: Per-role timeslot blocking — Hosted/General Buyer = 0, Sponsored TM = 4, Free TM / Participant = 12; admin impersonation is unlimited. [EP-45790]
  • RULE: Two-stage voting; a decline requires a reason ("Expired" on timeout). Meeting statuses Pending / Accepted / Declined / Expired / Cancelled (distinct from the self-managed meeting state machine). [EP-41245]
  • RULE: Stand-attendance booking (Buyer → Exhibitor company) is auto-confirmed and non-blocking. No-Show marking is available to both admin and user. Reserved "Women Circle" tables are supported. [EP-49073, EP-48314, EP-43263]

Round Tables

  • ENGINE: 3-level Session → Topics → Tables. "Max number of tables" (default 0 = unlimited) locks after Registration Ends; overflow applicants get status "Unassigned" and their card is silently removed from the schedule after the Final Schedule. Table numbering is continuous across topics with smallest-missing-number reuse. [EP-28986, EP-39819]
  • RULE: Optional Confirmation Process is one-time; a topic reassignment resets status to Pending (tagged System vs Organiser). Tables remain editable from session start until session end. [EP-41016, EP-46474]

Auto-approval algorithm (Exhibitor Events)

  • ENGINE: On-demand batch algorithm (Sessions Config → "Initiate auto-approval algorithm") maximises total approvals, guarantees every applicant ≥1 approval where possible, never approves overlapping events for one user, and balances approvals across users; under-subscribed events approve all. It is re-runnable — algorithm-rejected users may be approved on a later run; MANUALLY rejected users stay rejected. Events without a limit, On-Demand types, and Rejected / Rejected-for-Security-Concern statuses are excluded. [EP-46110, EP-46204]

19.3 Email / SES Pipeline

  • ENGINE: All email flows share a single AWS SES pipeline with a platform-wide bounce kill-switch keyed on aggregate bounce rate — one noisy client can pause delivery for ALL clients. A per-client transactional/marketing split is in rollout. [EP-53519]
  • RULE: Mass send-out supports >1000 recipients by segmentation, delivering to each recipient exactly once (no duplicates). [EP-35350]
  • RULE: Offensive-content report emails to the organiser are BATCHED every 15 minutes (no email if there were no reports in the window). [726990849]
  • RULE: Delivery language resolution — WEB uses the user's profile language (fallback: current browsing language); APP uses the account language; unconfigured falls back to the event default. Already-sent notifications are never re-translated. [EP-31011]
  • RULE: Email autologin — clicking a recommended card (or the Activation Link) applies autologin and lands the user on the frontend profile; the Activation Link also activates the account and must not work if no profile exists. [Part 12.1, EP-27950]

19.4 Badge-Scan Resolution

  • ENGINE: Scan resolution is fast-path/slow-path: the app first resolves the barcode to an account ID locally; on miss it sends the raw barcode to the backend, which searches barcode types barcode_new, external_qr, external_barcode, external_code. [Part 15.2]
  • RULE: ALL scans (recognised and unrecognised) are stored server-side and later reconciled against the event database; offline lead edits persist locally until a "Force scans sync". [EP-21095, EP-23400]
  • RULE: A lead is created on autosave even with partial or NO data; network loss saves locally and pushes on reconnection. [EP-47877]
  • EDGE CASE — barcode collision: if a barcode value equals another user's Account ID, the scan attributes to the WRONG person. [Part 15.2]
  • EDGE CASE — Account ID overflow: the maximum Account ID is 2,147,483,647 (signed 32-bit integer); barcodes exceeding this cause overflow errors. [Part 15.2]
  • RULE: OCR badge matching is exact 100% on First Name, Last Name, Job Title, Company; local DB first then server; multiple matches surface the top 5. A minimum of 3 extracted lines is required or the scan fails. OCR is lead scanning only, never login. [EP-45778]

19.5 Floor-Plan Rendering and Sync

  • ENGINE: A single shared floor-plan JS bundle (deployed to S3) is consumed by the frontend UI, admin panel and mobile apps; parent components layer background / stands / events. [EP-19626]
  • ENGINE: The mobile native floor plan is a webview of {front url}/newfront/widgets/floorplan. [EP-6677]
  • ENGINE: The renderer was rewritten in three.js (replacing react-konva) for performance at high stand counts, laying groundwork for a 3D view. [EP-18007]
  • RULE: ExpoFP sync — every exhibitor create/edit/delete pushes to ExpoFP when "Transmit exhibitor data to ExpoFP" is ON; changes are logged to CloudWatch; a resynchronization button covers exhibitors/categories/halls/stands with real-time status; API-token/credentials are validated with informative errors. [EP-26327, Mar–Apr 2025 Release Notes]
  • RULE: ExpoFP offline support downloads versioned ZIP archives ({expoKey}_{version}.zip) cached on-device (base URL app.expofp.com, build-status polling), with fallback to cache when online requests fail. [EP-45052]
  • RULE: Favourites ↔ ExpoFP stand bookmarks sync BI-DIRECTIONALLY; one stand can link to multiple exhibitors, and an exhibitor unfavourite removes the stand bookmark only if no other exhibitor on that stand is still favourited. [EP-44186]

19.9 Counting and Enforcement Rules (engine summary)

  • RULE: Meeting PENDING limits count against the INITIATOR; CONFIRMATION limits count against the RECEIVER. With autoconfirm ON, pending limits are not applied (nothing is ever pending), but the confirmation limit is still counted against the receiver — and category confirmation limits are enforced even under auto-confirm (auto-confirm stops at the limit). [EP-24046, 767885647, 1462075393]
  • RULE: Chat limits POOL at the exhibitor entity — company account + all team members share one pool — UNLESS a team member's participant badge category sets its own limits, which then apply instead. [697401523]
  • RULE: Rescheduling a meeting (pending or confirmed) never re-applies category limits. [EP-24046]
  • RULE: Reaching a pending limit HIDES the corresponding location type (online/offline) from the meeting form. Online minutes-credit exhaustion does NOT block creating online meetings — it removes "Join room" and ejects the user mid-meeting. [EP-24046]
  • RULE (runtime, Meet button): On WEB the availability chain runs ON CLICK (button not pre-hidden; target <1s; generic failure message shown); on the APP the same chain HIDES the Meeting button when no team member passes. Same absence-vs-error symptom, same root cause. [EP-35513, EP-37473]
